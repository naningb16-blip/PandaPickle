OVERVIEW:
---------
Cross-Site Request Forgery (CSRF) protection has been successfully implemented 
across the entire PandaPickle system. This prevents attackers from tricking 
users into performing unwanted actions.

HOW IT WORKS:
-------------
1. A unique token is generated per session when the user first visits the site
2. The token is stored in $_SESSION['csrf_token']
3. Every POST form includes a hidden input field with this token
4. When the form is submitted, the server verifies the token matches
5. If the token is missing or incorrect, the request is rejected with a 403 error

FUNCTIONS ADDED (includes/functions.php):
------------------------------------------
✅ generateCSRFToken() - Generates or retrieves the session CSRF token
✅ verifyCSRFToken($token) - Verifies the provided token matches session token
✅ csrfField() - Outputs hidden input field: <input type="hidden" name="csrf_token" value="...">
✅ checkCSRF() - Middleware function to verify POST requests have valid CSRF token

USAGE FOR FUTURE DEVELOPMENT:
------------------------------
When creating NEW POST forms, follow these steps:

1. Add CSRF field to the form:
   <form method="POST">
       <?= csrfField() ?>
       <!-- rest of form fields -->
   </form>

2. Add CSRF check at the top of POST handler:
   if ($_SERVER['REQUEST_METHOD'] === 'POST') {
       checkCSRF();
       // process form data
   }

ERROR HANDLING:
---------------
If CSRF validation fails:
- HTTP 403 Forbidden status is sent
- User sees: "CSRF token validation failed. Please refresh the page and try again."
- The request is immediately terminated with die()

SECURITY BENEFITS:
------------------
✅ Prevents attackers from forging requests on behalf of logged-in users
✅ Protects against one-click attacks via malicious links
✅ Uses cryptographically secure random tokens (bin2hex(random_bytes(32)))
✅ Uses hash_equals() for timing-attack resistant comparison
✅ Token is tied to user session - each user has their own unique token

NOTES:
------
- CSRF tokens are generated lazily (only when first needed)
- Tokens persist for the entire session duration
- Users with expired sessions will need to login again
- GET requests are NOT checked (they should not modify data)
- CSRF protection is independent of database (works with both MySQL and PostgreSQL)
